# HoopIQ — MVP-спецификация (B2B для тренеров)

## 1. Контекст в двух абзацах

HoopIQ перестраивается из приложения «для всех» в **инструмент тренера для развития игрового IQ команды**. Тренер назначает игрокам теоретический материал и квизы по реальным игровым моментам, игроки проходят их, тренер видит, кто прошёл и с каким результатом. Геймификация (очки, уровни, рейтинг) удерживает игроков и создаёт соревнование внутри команды.

## 2. Сердцебиение продукта (главный цикл)

Весь MVP строится вокруг одного повторяющегося цикла. Если он работает — продукт работает.

```
Тренер назначил  →  Игрок прошёл  →  Тренер увидел результат  →  Тренер скорректировал
       ↑                                                                      |
       └──────────────────────────────────────────────────────────────────────┘
```

MVP должен делать этот цикл возможным с минимумом функций. Ничего сверх цикла в MVP не входит.

---

## 3. Сущности (модель данных)

Три пользовательских сущности и два ключевых объекта.

### Пользователи
- **Тренер (Coach)** — создаёт команды, назначает задания, смотрит результаты.
- **Игрок (Player)** — проходит назначенное, копит очки, видит свой рейтинг.

### Объекты
- **Команда (Team)** — группа игроков под одним тренером.
- **Материал (Material)** — единица контента из библиотеки (набор обучающих карточек ИЛИ квиз). Создаётся владельцем продукта, не тренером.
- **Задание (Assignment)** — то, что тренер отправляет команде. Это и есть «гибкая активность тренера», оформленная как объект.

### Поля и связи

| Сущность | Ключевые поля |
|---|---|
| Coach | id, имя, email, [список команд] |
| Team | id, название, coach_id, invite_code, [список игроков] |
| Player | id, имя, team_id, очки, уровень |
| Material | id, тип (карточки / квиз), заголовок, тема, сложность, контент |
| Assignment | id, coach_id, team_id, [target_player_ids — пусто = вся команда], [material_ids], дедлайн, дата создания |
| Result | id, player_id, assignment_id, material_id, балл, дата прохождения |

**Связи:** Тренер 1→много Команд. Команда 1→много Игроков. Тренер создаёт Задания на Команду (опционально — на конкретных игроков). Задание ссылается на один или несколько Материалов. Прохождение игроком фиксируется как Result — именно он питает экран результатов у тренера и очки у игрока.

**Упрощения MVP:** игрок принадлежит ровно одной команде. Очки и уровень считаются от суммы баллов за пройденные материалы. Рейтинг = игроки команды, отсортированные по очкам.

---

## 4. Контент в MVP: только выбор из библиотеки

В MVP тренер **не создаёт** материал. Он только выбирает готовый из библиотеки, которую наполняет владелец продукта.

Почему так:
- даёт ценность с первого дня без усилий тренера;
- не перекладывает на тренера бремя создания контента (на нём отвалится большинство);
- авторство материала тренером — это отдельная большая функция и差ифференциатор, но строить его до валидации рано.

Библиотека на старте — это уже существующий контент HoopIQ: обучающие карточки и квизы по игровым моментам, размеченные по теме и сложности.

## 5. Объём MVP по ролям

### Тренер — возможности
- Регистрация, создание одной или нескольких команд.
- Приглашение игроков по коду/ссылке; добавление и удаление из состава.
- Просмотр библиотеки материалов.
- Создание Задания: выбрать материал(ы) → назначить команде или конкретным игрокам → задать дедлайн.
- Просмотр статуса задания: кто прошёл, кто нет, с каким баллом.
- Просмотр рейтинга команды.

### Тренер — страницы
- **Состав** — список команд и игроков, код приглашения.
- **Библиотека материалов** — просмотр и выбор контента.
- **Конструктор задания** — выбор материала, получателей, дедлайна.
- **Задания** — список созданных заданий со статусом выполнения.
- **Рейтинг команды**.

### Игрок — возможности
- Вход в команду по коду, создание профиля.
- Просмотр назначенного (что нужно пройти и до какого срока).
- Прохождение материала (существующий опыт HoopIQ: карточки-презентация и квизы).
- Начисление очков → рост уровня.
- Просмотр своего места в рейтинге.

### Игрок — страницы
- **Назначено мне** — главная страница: что в работе и что выполнено, с дедлайнами.
- **Материал/квиз** — существующий плеер прохождения.
- **Профиль** — очки, уровень, место в рейтинге.
- **Рейтинг команды**.

---

## 6. Ключевые сценарии MVP

1. **Настройка тренером.** Тренер регистрируется → создаёт команду → получает код приглашения → раздаёт игрокам.
2. **Вход игрока.** Игрок открывает ссылку / вводит код → создаёт профиль → попадает на «Назначено мне» (пока пусто).
3. **Назначение.** Тренер открывает библиотеку → выбирает материал → выбирает команду → ставит дедлайн → отправляет.
4. **Прохождение.** Игрок открывает задание → проходит карточки/квиз → получает очки → видит обновление рейтинга.
5. **Проверка.** Тренер открывает задание → видит, кто прошёл и с каким баллом → действует в реальности (правит тренировку / даёт следующее задание).

---

## 7. Вне MVP (намеренно отложено)

Чтобы не расползалось — это **не** в первой версии:
- создание и сборка материала тренером (Assemble / Create);
- аналитика по темам и «слабые места команды»;
- комментарии и фидбек тренера на ответы игрока;
- разбор соперника / playbook / подготовка к игре;
- объявления и заметки команде;
- системные уведомления и напоминания (в пилоте тренер оповещает игроков лично);
- сущности «Родитель» и «Админ школы»;
- рейтинги между командами/школами;
- свободный доступ игрока к библиотеке сверх заданий.

> Примечание по структуре данных: «Родитель» и «Админ школы» в MVP не делаем, но модель проектируем так, чтобы их добавление потом не требовало переписывать фундамент.
